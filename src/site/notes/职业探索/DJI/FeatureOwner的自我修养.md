# FO的能力画像

评价一个人是否能承担FO的维度如下：

|维度|阐释|个人评价|
|---|---|---|
|**Owner意识，追求卓越的产品（根本动力）**|• 产品侧：积极地体验自己的功能，寻找可以改进的点，**不断打磨，不做轻易妥协；**  <br>• 交付侧：交付过程中**不断主动思考**存在的问题并加以解决；|在产品侧容易妥协|
|**扎实的业务、技术能力**|• 本模块：对**本模块业务**熟悉，可以独立设计优秀的技术方案；  <br>• 跨端：知道各端、各模块之间如何配合的，能画**系统框架**图；  <br>• **产品：**能提出产品意见，时刻思考是否真的解决了用户问题；  <br>• **质量：**能和质量同事打合清除测试方案和计划；|对前后端技术可以多了解|
|**高效的沟通协作能力**|• **信息传递，**清晰、及时地传递计划、进展、问题，确保相关人都理解；  <br>• **协调共赢**，积极应对各方意见给出合理反馈，**避免发生冲突**，能够取得**成员的信赖**；|• 有时候计划传递不够及时全面  <br>• 辩论能力还有待提高  <br>• 成员调动上欠缺方法|
|风险上报与应对能力|• 能够**识别风险**，及时通过合理途径**上报风险；**  <br>• 能够讲清风险的**应对策略**，根据风险**灵活调整计划**，持续跟进**闭环风险**；|• 偏乐观主义  <br>• 风险跟进闭环有拖延症|
|交付流程认知与经验|• 掌握科学的**交付管理方法**，详细见”FO的职责方法“  <br>• 能够多反思，多**复盘总结经验**|做得不错|

对于我自己来说，最欠缺也是认为最重要的能力：**风险的识别与及时跟进能力**。

## 需求来源

为什么要了解需求的来源？主要是因为需求从用户传递到开发这个过程中是有信息错损的。为了避免需求偏离、遗漏的情况，需要知道需求是谁提出的，并且在需求设计、开发阶段反复对齐确认。

其次，对于需求的来源方，在提出后并不是不管不顾了，而是非常期望能够得到FeatureOwner关于需求开发进展的反馈，否则需求来源方对FeatureOwner会充满不确定感、逐渐失去信心，即使这次需求按时交付了，下次可能也不会找他了。

  

## 需求边界

需求边界很容易被想当然地忽视或者低估。需求边界决定了FO开展管理的范围，特别是涉及多模块、多业务组的需求，FO很容易陷入只关注内部的境况，从而导致资源、方案评估偏乐观，遗漏了产品设计、开发工作量和测试用例范围。

  

## 需求的核心矛盾

把握一个事物要抓其主要矛盾的主要方面。尤其对于一个大的需求而言，由于资源的局限，我们不可能面面俱到，所以必须搞清楚什么对用户是最有价值的，并集中力量去实现价值最高的部分。因此，在需求的第一版交付中，应该始终围绕核心矛盾来做，并在第一版发布后验证是否解决了用户的痛点，即真正的核心矛盾是否得以解决。在后续的版本中，才去考虑那些次要的事情。

  

## 计划倒排现象

在计划输出阶段，经常出现准出节点与计划不匹配的情况。对此而采用的计划倒排是不科学的，因为不匹配无非是以下原因：

- 需求工作量太大，排不下；

- 需求估时太水或太保守；

无论是那种原因，本质都反映了风险的存在。而计划倒排不过是短暂的逃避风险罢了。正确的做法应该是：

- review ROI与需求核心矛盾，将不必要的功能提早砍掉，只做收益最大的部分；

- review 方案合理性，是否存在简单方案快速实现的可能；

- 如果发现方案是合理的，只是估时太水，这个时候应当注意开发者态度与能力的风险，并在开发过程中重点关注；

最后，从现实经验来看，很多估时其实只会少不会多，这是因为一方面大多的人都是偏乐观的；另一方面在后续需求开展过程中一定存在遗漏的额外工作。

  

# 讲清目标与现状

## 目标是什么

必须让成员真正清楚目标，最差的情况是只有管理者知道目标。包括以下维度。可以制作一张需求-时间二维时间表用来跟进。

|   |   |
|---|---|
|需求的标准（WHAT）|• 明确有哪些需求  <br>• 明确需求做成什么样是合格的|
|对谁提要求（WHO）|• 当团队成员都具备必要能力时，可以采用简单的管理方式；而当团队成员存在一个能力较差的人时，就需要专门针对其进行辅导，讲清对他的要求，并时常闭环|
|明确时间节点（WHEN）|• 需求什么时候联调、什么时候准出（发dev）、什么时候提供自测结果|

# FO的个人经验总结

|主题|案例|总结|疑惑|
|---|---|---|---|
|需求背景评估|进离场航线需求中，飞控模块提了一个关闭智能返航模式的要求，一开始没有仔细分析就接受了，但后来仔细盘算才发现这个东西不合理：  <br>• 需求是飞控FO单方面提出的；  <br>• 是为了解决用户开启智能返航导致不按进立场返航有炸鸡风险，但是即使关闭了也有其他的途径会退出进离场返航，所以无法完全解决问题；|对于需求过程中提出的小需求也需要遵循“事前”方法，评估需求的背景价值。|这种是否走需求变更？经常有在产品宣导后开发了解到要修改产品方案的时候，走什么流程？|
|方案合理性|进离场航线上传方案涂省事没有仔细评估方案的正向、合理性，导致最后返工|方案要正向且完备||
|秉持公心、维护周边关系|进离场航线需求中，为了推脱进离场临时导入方案的工作以及行服交付时间不匹配问题，多次拉项目组和飞控争论KOC支持方案，最后却发现：  <br>• 从司空导入是工作量最小的，并且方便了后期远程调试。  <br>• 过渡掰扯破坏了兄弟模块关系，浪费了项目组大哥时间|考虑大局和整体，不要盲目甩锅，追求最优解||
|风险识别与及时闭环|• 进离场航线上传方案风险爆出后，拖了两三个月才开始重视解决  <br>• 飞行记录开放方案一直处于待办状态但没有跟进，最后发现开放方案变更了  <br>• 指令飞行的影响一直让Otto导单子，但是实际没有拉远程操控组进行对齐修改点，导致远程操控组排期问题  <br>• 航线上传、下载方三端方案一直没有明确过，到了快开发的时候才找IOT，这个时候人家都没有人力搞这个  <br>• 飞行记录二期时间本来就因为一期delay紧张，面对产品提的优化需求，没有再一开始就识别出来做不了|切忌乐观主义，及时消除不确定性，有风险第一时间解决||
|长线需求的技术方案管理|飞行记录二期，在空中航线下发 、媒体地图定位开发过程中才发现后端需要补充字段；一期接口方案是一次性设计的，二期的功能点并没有详细评估过||长线需求，拆解成一个个功能点后，技术方案节奏怎么展开？  <br>【方式一】在一开始就把by功能点一一把关键功能的方案评审了，但是实际行不通：对需求的认知欠缺，很难起初就考虑得全面，后面还得调；  <br>【方式二】在每一期开始对本期内容进行方案评审，这个感觉还稍微行得通，就像行服每个版本一样：1. 要求在二期开始前，产品要将一期功能产品方案宣讲都解决掉，对时间点的要求高 2.每一期开始要留两天时间集中搞方案；  <br>【方式三】做每个功能点前方案评审：时间太晚了，对原计划的冲击大，会造成计划频繁更改；  <br>其他问题：  <br>1. 功能点的评审方式和平台是啥？总的方案是上技术架构会，单点功能多是接口设计，目前好像没有这个界面？  <br>2. 长线需求时间调整感觉是必然的？那么对delay时间也有个预期的容忍度，这个容忍度是否要纳入到准出时间制定上？|
